# analyze-image bundled extension

> ⚠️ **Это НЕ замена vision-моделям.**
>
> Этот extension — минимальный костыль, чтобы text-only модели получили хоть какую-то
> информацию об изображениях. Он **не понимает** визуальный контекст, композицию, стиль,
> эмоции, жесты, графики, иконки, UI-макеты, таблицы без captioning.
> Если у модели есть поддержка vision (Claude 3.5, GPT-4o, Gemini и т.д.) —
> используйте её, а не этот tool.

Даёт text-only LLM базовую информацию об изображении: распознанный текст (OCR),
метаданные (размер, формат) и опциональное описание (captioning).
Никаких пикселей в LLM не передаётся — только структурированный текст.

## Что extension умеет (и не умеет)

| Возможность | Статус | Ограничения |
|-------------|--------|-------------|
| OCR (Tesseract) | ✅ | Только печатный текст. Не читает рукописный, мелкий (<10px), искажённый. Зависит от traineddata для языка |
| Метаданные | ✅ | Размер, формат, прозрачность. Работает для всех распространённых форматов |
| Классификация (screenshot/photo/icon) | ✅ | Rule-based, не ML. Может ошибаться на маленьких или нестандартных изображениях |
| Captioning (vit-gpt2) | ⚠️ Опционально | **Очень низкое качество.** Модель обучена на COCO (фото котиков, людей, машин). На UI-скриншотах, графиках, мемах — выдаёт бессмыслицу. Не описывает кнопки, меню, попапы, оси, легенды. Инференз 3-5s на CPU, но первая загрузка модели из кэша ~10-20s |
| Цвета | ⚠️ | Приблизительные. Для JPEG не поддерживается (нужен sharp) |
| Понимание UI/UX | ❌ | Не знает что такое кнопка, меню, попап. Только текст + caption |
| Графики/диаграммы | ❌ | Не парсит оси, легенды, точки данных. OCR может прочитать подписи |
| Рукописный текст | ❌ | Tesseract не обучен на рукописном |

## Как это работает

Когда агент встречает изображение (screenshot, error dialog, UI mockup, документ),
он вызывает tool `analyze_image`. Tool:

1. Читает **метаданные** (размер, формат, тип) — мгновенно, без зависимостей
2. Запускает **OCR** (Tesseract.js или native CLI) — извлекает текст с координатами и confidence
3. **(Опционально)** **Rule-based классификация** — screenshot/photo/icon/animation
4. **(Опционально)** **Captioning** через `@huggingface/transformers` — описывает содержимое

Всё возвращается агенту как plain text.

## Когда использовать

✅ Когда модель не поддерживает vision:
- Ошибки в терминале (screenshot текста)
- Код на скриншоте
- Документы, PDF (как fallback)
- Короткие сообщения об ошибках

❌ Когда лучше использовать vision-модель:
- Сложные UI/UX макеты
- Фотографии (их нужно описывать визуально)
- Графики и диаграммы (OCR часто бессилен)
- Иллюстрации, мемы, жесты

## Зависимости

| Пакет | Обязателен? | Размер | Зачем |
|-------|-------------|--------|-------|
| `tesseract.js` | ✅ Да | ~5MB JS + traineddata | OCR |
| `@huggingface/transformers` | ❌ Нет (captioning) | ~3MB JS + ~245MB model | Image captioning |

## Установка

```bash
# В корне pi-mono-x (workspace)
npm install tesseract.js@^7.0.0 --workspace=packages/coding-agent
```

Для captioning (опционально, только на мощных машинах):
```bash
npm install @huggingface/transformers
# Первый запуск скачает модель ~245MB с HuggingFace Hub
```

## Использование

Агент вызывает tool автоматически, когда нужно прочитать текст из изображения:

```
analyze_image({
  image_path: "/tmp/screenshot.png",
  analyze: ["ocr"]  // или ["ocr", "caption"] для описания
})
```

**Важно:** Если модель поддерживает vision (`ctx.model.input.includes("image")`),
`input` handler НЕ трогает изображение — оно передаётся модели как есть.

## Конфигурация

Настройки в `~/.pi/agent/extensions/analyze-image/config.json`:

```json
{
  "ocr_enabled": true,
  "ocr_lang": "eng+rus",
  "captioning_enabled": false,
  "captioning_backend": "tiny",
  "rule_based_classification": true,
  "max_image_size_mb": 10
}
```

По умолчанию captioning выключен — только OCR + metadata.
Captioning включается в Settings → Image Analysis, либо через параметр `analyze: ["caption"]`.
